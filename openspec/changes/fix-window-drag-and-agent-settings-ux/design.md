## Context

当前共享标题栏 `WindowTitlebar` 的最外层容器使用了 `titlebar-drag`，但其内部左侧 breadcrumb、中心 actions、右侧窗口控制三块可见内容都被标记成了 `titlebar-no-drag`。在 `WorkPage` 这类页面中，标题栏视觉上几乎被这些内部区域完全占满，导致用户实际上只能在很窄的空隙上拖动窗口。这个问题不是单页偶发，而是共享页面 chrome 命中区过小的系统性问题。

AI 助手设置弹窗 `AgentSettingsPanel` 已经具备测试连接与保存逻辑，但当前交互仍存在三类缺口：一是动作区在受限视口或嵌入式聊天容器中不够稳固，用户容易感知为“没有保存/验证按钮”；二是保存闭环需要更明确地围绕当前草稿值、快捷键注册结果和持久化结果来组织；三是弹窗还不支持桌面应用常见的 backdrop dismiss，而且关闭触发点需要落在鼠标释放阶段，避免按下时误关闭。

## Goals / Non-Goals

**Goals:**
- 让共享标题栏为 frameless 主窗口提供稳定、易命中的拖拽区域，而不是依赖零碎空白区域。
- 保留标题栏内返回、同步、登出、窗口控制等交互按钮的可点击性，不把它们变成拖拽热点。
- 让 AI 助手设置弹窗的测试连接与保存动作在常见窗口高度下始终可见、可触达，并使用当前草稿值完成验证和保存。
- 让 AI 助手设置在保存后真正保留配置，并在快捷键局部失败时给出明确反馈而不是静默丢失。
- 为 AI 助手设置弹窗增加符合桌面直觉的外部点击关闭，并把判定放在鼠标释放阶段。

**Non-Goals:**
- 不重做整个窗口视觉设计，也不替换当前 frameless BrowserWindow 方案。
- 不新增 AI provider 协议或改变现有 workflow planner 的业务语义。
- 不扩展成通用 dialog 框架重构；本次仅聚焦 AI 助手设置弹窗与共享标题栏。

## Decisions

### 1. 为共享标题栏引入显式的拖拽命中区，而不是继续依赖“按钮之间剩余空白”

标题栏应当保留一块明确的 drag-safe 区域，例如让外层标题栏承担拖拽语义，并确保至少有一段横向可见区域不被 `titlebar-no-drag` 内容完全占满；必要时可通过布局占位、独立拖拽层或拖拽条来放大命中面积。这样可以让用户从页面上边缘直接拖动，而不必精确寻找小缝隙。

选择原因：当前问题的根源是命中区设计，而不是 Electron 自身无法拖动。只改样式或增大高度但不重分配 drag/no-drag 区域，仍然会保留“看得到、拖不到”的问题。

备选方案：简单移除左侧区域上的 `titlebar-no-drag`。未采用，因为 breadcrumb、返回按钮和中间 actions 仍需稳定点击，粗暴放开会引入误拖拽。

### 2. 将 AI 设置弹窗的动作区视为固定底部 chrome，而不是普通滚动内容的一部分

弹窗内容体保持滚动，但“测试连接 / 保存 / 取消”动作区应固定在卡片底部，配合更稳妥的最大高度与滚动区域尺寸计算，避免在较小宿主窗口中被折叠或推到不可见位置。必要时可增加 sticky footer 或显式的 body/footer 分层。

选择原因：用户感知“没有验证和保存按钮”通常不是逻辑完全缺失，而是按钮位置不稳定、超出可见区或不够显眼。把动作区上升为 modal chrome 能直接解决这一类问题。

备选方案：继续让整张卡片随内容一起滚动。未采用，因为表单一长，关键动作会再次沉到底部，问题会复发。

### 3. 保存与验证都以当前草稿态为准，并把保存结果分为“配置成功”和“快捷键附带状态”两个层次

测试连接应使用当前输入框里的未保存草稿值；保存流程应先整理出标准化配置，再提交共享 store 与快捷键设置，并明确区分“AI 配置已保存”与“快捷键注册失败但配置仍已保存”两种结果。弹窗只应在主保存路径成功后关闭，失败时保留现场供用户继续调整。

选择原因：当前用户反馈的重点是“配完不能保存”，所以设计必须突出真正的持久化闭环，而不是只执行本地状态更新。

备选方案：点保存后无论结果如何立即关闭。未采用，因为这会让失败路径几乎不可见，用户只能以为设置没生效。

### 4. Backdrop dismiss 采用 pointer down/up 配对，而不是简单 backdrop onClick

弹窗外部关闭应记录指针是否从 backdrop 开始，并仅在同一次交互最终于卡片外区域释放时触发关闭；如果按下后把鼠标移回卡片内松开，或交互发生在卡片内部，则不关闭。这样能严格满足“不是按下时关闭，而是松开时且落点在卡外才关闭”的要求。

选择原因：普通 `onClick` 虽然看似简单，但在拖拽、文本选择、按下后移入/移出等场景里语义不够明确，不利于得到稳定的桌面手感。

备选方案：直接在 backdrop 的 `onMouseDown` 关闭。未采用，因为这正是用户明确不想要的行为。

## Risks / Trade-offs

- [标题栏拖拽区扩大后误吞按钮点击] → 通过明确的 `titlebar-no-drag` 控件区和专用拖拽区隔离热点。
- [固定底部动作区会压缩表单可见高度] → 保持表单体滚动，并为 footer 预留稳定高度而不是让整体溢出。
- [backdrop 指针配对增加交互状态复杂度] → 仅跟踪主按钮指针，关闭条件限制为“按下和释放都发生在同一 backdrop 交互链路”。
- [保存流程显式暴露部分失败状态后，用户会看到更多提示] → 将提示分为成功、部分成功、失败三个层级，减少歧义。

## Migration Plan

- 无需数据库 schema 迁移；现有 AI 配置与快捷键设置继续沿用当前持久化结构。
- 共享标题栏调整为纯 renderer 布局/命中区优化，可直接替换现有组件实现。
- 为 AI 设置弹窗补充回归测试，覆盖按钮可见性、保存持久化、验证使用草稿值和 backdrop dismiss 语义。
- 若交互调整引入意外误关闭或误拖拽，可独立回退对应 renderer 逻辑，不影响底层数据。

## Open Questions

- AI 助手设置保存成功后是否还需要额外 toast / inline success 文案，还是关闭弹窗即可满足反馈需求？
- 迷你助手未来若需要独立设置入口，是否复用同一 modal 行为契约即可，还是要再拆分尺寸与宿主策略？
